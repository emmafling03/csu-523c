Model Building

Split the Data
  1. Set a seed for reproducibility
  2. Transform the q_mean column to a log scale. Remember it is error prone to apply transformations to the outcome variable within a recipe.
  3. Split the data into a training and testing set.
    1. Use 80% of the data for training and 20% for testing.
    2. Leave stratification off in this example, but stratified splits often matter when the outcome has structure
  4. Create a 10-fold cross validation dataset to help evaluate multi-model setups.
set.seed(123)
# Bad form to perform simple transformations on the outcome variable within a 
# recipe. So, we'll do it here.
camels <- camels |> 
  mutate(logQmean = log(q_mean))

# Generate the split
camels_split <- initial_split(camels, prop = 0.8)
camels_train <- training(camels_split)
camels_test  <- testing(camels_split)

camels_cv <- vfold_cv(camels_train, v = 10)
Preprocessor: recipe

recipe() lets us declare a preprocessing pipeline once, estimate its parameters on the training data, and reuse it everywhere: folds, test set, new predictions.

Known:

  • q_mean, aridity, and p_mean columns are right skewed and can be helped by log transformations

  • the relationship between aridity and p_mean is non-linear and can be helped by adding an interaction term.

Build a small recipe that encodes those decisions

# Create a recipe to preprocess the data
rec <-  recipe(logQmean ~ aridity + p_mean, data = camels_train) %>%
  # Log transform the predictor variables (aridity and p_mean)
  step_log(all_predictors()) %>%
  # Add an interaction term between aridity and p_mean
  step_interact(terms = ~ aridity:p_mean) |> 
  # Drop any rows with missing values in the pred
  step_naomit(all_predictors(), all_outcomes())

Where things get a little messy

Wrong version 1: augment

The broom package provides a convenient way to extract model predictions and residuals. We can use the augment function to add predicted values to the test data. However, if we use augment directly on the test data, we will get incorrect results because the preprocessing steps defined in the recipe object have not been applied to the test data.

Correct version: prep -> bake -> predict

To correctly evaluate the model on the test data, we need to apply the same preprocessing steps to the test data that we applied to the training data. We can do this using the prep and bake functions with the recipe object. This ensures the test data is transformed in the same way as the training data before making predictions.

test_data <-  bake(prep(rec), new_data = camels_test)
test_data$lm_pred <- predict(lm_base, newdata = test_data)

Using a workflow instead

tidymodels provides a framework for building and evaluating models using a consistent and modular workflow. The workflows package lets you bundle a preprocessor (recipe or formula) and a model spec into a single object, then fit them together. This makes it easier to experiment with different models, compare performance, and ensure reproducibility.

Here, we use linear_reg() to define a linear regression model, set the engine to lm, and the mode to regression. We then add our recipe to the workflow, fit the model to the training data, and extract the coefficients.

# Define model
lm_model <- linear_reg() %>%
  # define the engine
  set_engine("lm") %>%
  # define the mode
  set_mode("regression")

# Instantiate a workflow ...
lm_wf <- workflow() %>%
  # Add the recipe
  add_recipe(rec) %>%
  # Add the model
  add_model(lm_model) %>%
  # Fit the model to the training data
  fit(data = camels_train) 

# Extract the model coefficients from the workflow

Workflow Set

workflow_set is a powerful tool for comparing multiple models on the same data. It allows you to define a set of workflows, fit them to the same data, and evaluate their performance using a common metric. Here, we are going to create a workflow_set object with the linear regression and random forest models, fit them to the training data, and compare their performance using the autoplot and rank_results functions.

wf <- workflow_set(list(rec), list(lm_model, rf_model)) %>%
  workflow_map('fit_resamples', resamples = camels_cv) 

autoplot(wf)
rank_results(wf, rank_metric = "rsq", select_best = TRUE)

Evaluation

A proper evaluation has three parts:

  1. Variable importance — which predictors is the model actually using? Week 5-2 introduced vip::vip() paired with extract_fit_parsnip() to pull the underlying model object out of a fitted workflow.

  2. Test-set metrics — apply the trained model to data it has never seen.

  3. A predicted-vs-observed plot — the visual sanity check; the 1:1 line should run through the cloud.

Recipe

Formula

logQmean ~ aridity + p_mean + pet_mean + elev_mean + slope_mean +
           soil_porosity + soil_conductivity + max_water_content +
           frac_snow + lai_max + frac_forest
logQmean ~ aridity + p_mean + pet_mean + elev_mean + slope_mean + 
    soil_porosity + soil_conductivity + max_water_content + frac_snow + 
    lai_max + frac_forest

The goal of the model is to predict mean streamflow, which is influenced by parameters such as climate, topography, soil, and vegetation characteristics. Variables such as precipitation and aridity are expected to be the strongest drivers of streamflow, as they directly control water and atmospheric demand. Topographic variables like elevation and slope influence runoff and routing. Soil properties such as porosity, conductivity, and maximum water content affect infiltration and storage. Leaf area index and forest fraction, impact evapotranspiration and water retention. According to the CAMELS pdf, these variables represent key hydrologic processes such as precipitation inputs, evapotranspiration demand, and watershed storage characteristics .

Exploratory data analysis showed that q_mean and several predictors are right-skewed, justifying the use of log transformations. Additionally, the relationship between aridity and precipitation is non-linear, motivating the inclusion of an interaction term. Missing values are minimal but present, so removing incomplete cases ensures consistency in model training. Since predictors are on different scales, so normalization is used to improve model stability and performance.

Recipe

rec <- recipe(
  logQmean ~ aridity + p_mean + pet_mean + elev_mean + slope_mean +
    soil_porosity + soil_conductivity + max_water_content +
    frac_snow + lai_max + frac_forest,
  data = camels_train
) %>%
  step_naomit(all_predictors(), all_outcomes()) %>%
  step_log(aridity, p_mean, pet_mean, frac_snow, offset = 1) %>%
  step_interact(terms = ~ aridity:p_mean) %>%
  step_normalize(all_predictors())

The variable importance plot shows that the tuned random forest relied most heavily on p_mean and aridity, with a large decrease to predictors. The CAMELS pdf defines p_mean as mean daily precipitation, pet_mean and aridity as measures of atmospheric demand, frac_snow as the fraction of precipitation falling as snow, elev_mean and slope_mean as topographic descriptors, and lai_max and forest fraction as vegetation characteristics, so the VIP ranking is physically reasonable.

Comparing to the Moran’s I shows that several important predictors are also spatially autocorrelated. For example, slope_mean, p_mean, elev_mean, and lai_max are more spatially structured predictors in the Moran’s I table, meaning part of their predictive power may come from regional climate patterns and direct hydrologic mechanisms. The most extreme spatial variables in the Moran’s I results are gauge_long and gauge_lat, and were not the dominant VIP features in the final model, which is reassuring as this indicates they are geographic proxies only. When considering the relatively small difference from random CV to spatial CV, this suggests the model is using real environmental signal rather than relying geographic shortcuts, though some variables may reflect both mechanism and location.
